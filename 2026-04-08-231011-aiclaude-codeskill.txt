● AI时代的学习理论与 Claude Code Skill 结合

  核心学习理论

  1. Connectivism (CONNECTIVISM)

  George Siemens 提出——"学习是连接网络的过程"，而非知识单向灌输。这与 Claude
  Code 的 skill 系统天然契合：
  - Skill 本身就是知识网络的节点
  - 可以设计一个 skill 自动构建概念图谱（concept graph），把新主题与已有知识连接

  2. Deliberate Practice + Bloom's 2-Sigma

  Ericsson 的刻意练习理论 × Bloom 的 2-Sigma 问题（个性化 tutoring 能提升 2
  个标准差）：
  - AI 最大优势之一就是模拟一对一 tutoring
  - Skill 可以实现：苏格拉底式提问（Socratic
  questioning）——不直接给答案，引导你自己推导出答案
  - 结合 预测试效应（pre-testing）——先做题，再学，记忆更深刻

  3. Retrieval Practice / Testing Effect (提取练习)

  比重复阅读强得多的记忆方式：
  - 核心机制：先回忆，再学习（测试自己 → gap → 再学习）
  - Skill 层面：可以自动生成 flashcard 序列（基于 SM-Algorithm 或 FSRS 算法）

  4. Spaced Repetition (间隔重复)

  Ebbinghaus 遗忘曲线 → SM-Algorithm / FSRS：
  - Anki 原理：在你即将遗忘时复习
  - Claude Code skill 可以动态生成复习计划，配合项目实践

  5. Interleaving (交错练习)

  混合不同子主题练习，而非批量学完 A 再学 B：
  - Skill 可以把一个大主题拆成交错的子任务，强制你在子主题间切换

  6. Zettelkasten / Building Second Brain

  卡片式笔记法 + 知识图谱：
  - 每个概念一张卡片，带唯一 ID
  - 卡片间双向链接形成网络
  - Claude Code skill 可以帮你自动生成卡片 + 维护链接

  7. Pareto / 80-20 Principle

  先学最关键 20% 概念，覆盖 80% 场景：
  - Skill 可以分析任意主题，自动识别核心概念集（essential concepts）

  8. Metalearning (元学习)

  "学会如何学习"：
  - 这个 skill 本身就是一个 meta-skill——教你快速掌握任何主题的方法论

  ---
  Quick Mastery: Kubernetes Networking

  30-Second Summary

  Kubernetes networking solves one problem: how do Pods talk to each other
  across nodes, and how do external traffic reach Pods. The core model is: every
   Pod gets its own IP, containers share a network namespace, and Services
  provide stable entrypoints via label selectors. Everything else (DNS, Ingress,
   NetworkPolicies) builds on top of this.

  ---
  Essential Concepts (5-7)

  1. Pod Networking Model

  - What it does: Each Pod gets a unique IP, containers within a Pod share the
  same network namespace (localhost works between them)
  - Key insight: Think of a Pod as a "logical machine" — the IP is stable for
  the Pod's lifetime, not the container's
  - First step: kubectl get pod -o wide → 看到 IP 列就是 Pod IP

  2. Service (ClusterIP)

  - What it does: Stable virtual IP + DNS name that load-balances to matching
  Pods via label selector
  - Key insight: Service is a logical router — it never moves, Pods come and go,
   traffic just routes to healthy ones
  - First step: kubectl expose deployment my-app --port=80 → 创建 ClusterIP
  Service

  3. kube-dns / CoreDNS

  - What it does: DNS-based service discovery within the cluster
  - Key insight: my-svc.my-namespace.svc.cluster.local = the full DNS name;
  short name works within the same namespace
  - First step: nslookup my-svc from any Pod to verify DNS resolution

  4. Ingress

  - What it does: HTTP/HTTPS routing from outside the cluster to Services, with
  host/path rules
  - Key insight: Ingress controller (nginx, contour, etc.) watches Ingress
  resources and configures the actual proxy — without a controller, Ingress
  resource does nothing
  - First step: Deploy an Ingress controller first, then create Ingress rules
  pointing to your Service

  5. NetworkPolicies

  - What it does: Firewall rules at the Pod level — who can talk to whom
  - Key insight: Kubernetes is allow-by-default inside a namespace;
  NetworkPolicies restrict inbound/outbound traffic explicitly
  - First step: Start with kubectl label namespace my-ns name=my-ns then apply a
   default-deny policy before writing allow rules

  6. CNI (Container Network Interface)

  - What it does: Plugin system that actually assigns Pod IPs and sets up
  routing across nodes (Calico, Cilium, Flannel, Weave)
  - Key insight: CNI is the implementation — the Pod networking model is spec,
  CNI is the code. Wrong CNI = weird networking bugs
  - First step: kubectl get nodes -o wide → check POD-CIDR column to see CNI's
  IP allocation

  7. Service Types (ClusterIP / NodePort / LoadBalancer)

  - What it does: Three ways to expose a Service — internal only, via node port
  (30000-32767）, or via cloud LB
  - Key insight: They nest — NodePort includes ClusterIP, LoadBalancer includes
  NodePort. Choose the minimum scope you need
  - First step: my-svc.cluster.local:80 works for ClusterIP; need external? →
  NodePort first before cloud LB

  ---
  Quick Mastery: TypeScript Generics

  30-Second Summary

  TypeScript generics let you write code that works with any type while still
  preserving type safety — you define the shape once, and TypeScript infers the
  specific type when you use it. The core mental model: generics are to types
  what functions are to values. Just as a function abstracts over inputs, a
  generic abstracts over types. The payoff: reusable components that are as
  type-safe as handwritten non-generic code.

  ---
  Essential Concepts (5-7)

  1. Generic Functions

  - What it does: Write one function that works with multiple types, TypeScript
  infers at call site
  - Key insight: <T> is a type parameter — a variable for types, not values
  - First step: function identity<T>(x: T): T { return x } → identity(42) gives
  number, identity("hi") gives string

  2. Generic Interfaces & Types

  - What it does: Define reusable type shapes that work with any type parameter
  - Key insight: interface Box<T> { value: T } — the T is a placeholder filled
  in when you use it
  - First step: type Pair<A, B> = [A, B] → Pair<string, number> is a tuple
  [string, number]

  3. Constraints (extends)

  - What it does: Limit what types can be passed to a generic — require certain
  properties exist
  - Key insight: extends doesn't mean inheritance — it means "must be assignable
   to"
  - First step: function getLength<T extends { length: number }>(x: T): number →
   works on strings, arrays; fails on numbers

  4. keyof + Indexed Access Types

  - What it does: Extract keys from a type, then access those key types
  dynamically
  - Key insight: keyof T = union of all property names of T; T[K] = type of
  property K in T
  - First step: function getProperty<T, K extends keyof T>(obj: T, key: K): T[K]
   — return type varies based on which key

  5. Conditional Types (T extends U ? X : Y)

  - What it does: Type-level if/else — choose a type based on whether another
  type matches a constraint
  - Key insight: Evaluated at type-check time, not runtime — pure type machinery
  - First step: type IsArray<T> = T extends any[] ? true : false →
  IsArray<string[]> = true, IsArray<string> = false

  6. Utility Types (Mapped + Template Literals)

  - What it does: Built-in generics that transform existing types (Partial,
  Required, Pick, Omit, Record)
  - Key insight: Partial<T> = { [K in keyof T]?: T[K] } — every property becomes
   optional
  - First step: Partial<User>, Pick<User, "id" | "name">, Omit<User, "password">

  7. infer Keyword

  - What it does: Extract a type from within a conditional type — "infer this
  part for me"
  - Key insight: Only valid inside the extends clause of a conditional type
  - First step: type ReturnType<T> = T extends (...args: any[]) => infer R ? R :
   never → extracts function's return type

  Flashcards

  Q1: What's the difference between <T> and <T extends Constraint>?
  A1: <T> accepts ANY type; <T extends Constraint> only accepts types that have
  the required properties | cue: extends = "T must be at least X" — limits input
   types

  Q2: What does keyof T return?
  A2: A union of string/number literal types — all property names of T | cue:
  "keyof" = "what keys exist on" — like Object.keys() but at type level

  Q3: Why use Pick<T, K> instead of just writing the picked type manually?
  A3: Pick stays in sync with T — if T adds a property, Pick<T, "id"> still
  works. Manual writing drifts and becomes stale | cue: Pick is "live" not
  "baked"

  Q4: When does infer fire?
  A4: Only inside the extends clause of a conditional type, only when the
  condition is true | cue: infer = "I see this pattern, extract X from it"

  Q5: Generic constraint says T extends { length: number }. What passes? What
  fails?
  A5: Pass: string, array, { length: 5 }. Fail: number, boolean, { size: number
  } | cue: extends = "must have length: number property"
